# spec-requirements: EARS形式 要件定義スキル

## このスキルの目的

プロジェクトの要件定義書(`doc/requirements.md`)を、EARS (Easy Approach to Requirements Syntax) 形式で生成・維持する。

このスキルが生成するのは**「何を実現するか」(What)**であり、**「どう実現するか」(How)**ではない。実装詳細、アーキテクチャ選定、データモデル設計はこのスキルの範囲外。要件定義の後工程として別途 domain-model.md が存在するが、それは別のスキルが担当する。

## なぜ否定的境界が重要か

このプロジェクト（サイクリングルート自動生成）は非常にニッチなドメインである。「ルート生成」「地図」「サイクリング」といったキーワードから、AIエージェントはナビゲーション機能、ソーシャル機能、リアルタイム交通情報、AI最適化など一般的なサービスの機能を連想して拡大解釈しやすい。

そのため、要件定義には2階層の否定的境界を設ける:
1. **プロジェクトレベルの Non-Goals**: プロジェクト全体としてやらないこと
2. **各要件の Out of Scope**: その要件が含まないこと

これにより、エージェントがスコープ外の機能を提案・実装するリスクを構造的に抑制する。

## 呼び出しパターン

| コマンド | 動作 |
|---|---|
| `/spec-requirements` | 対話を通じて初期要件定義書を生成する |
| `/spec-requirements add` | 既存の要件定義書に新しい要件を追加する |
| `/spec-requirements modify N` | 要件番号Nを修正する |
| `/spec-requirements review` | 既存の要件定義書の整合性・網羅性をレビューする |

## 要件定義書のフォーマット

生成する `doc/requirements.md` は以下の構造に従う:

```markdown
# 要件定義書: [プロジェクト名]

## プロジェクト概要

[プロジェクトが何をするものかを2-3文で記述]

## Non-Goals（プロジェクトスコープ外）

以下はこのプロジェクトの対象外である:

- [やらないこと1]
- [やらないこと2]
- ...

---

### Requirement 1: [機能名]

**Objective**: [役割]として、[機能]を実現したい。それにより[便益]が得られる。

**Acceptance Criteria**:
- When [イベント], the system shall [応答].
- While [状態], the system shall [応答].
- If [例外条件], the system shall [応答].
- Where [オプション条件], the system shall [応答].
- The system shall [普遍的な振る舞い].

**Out of Scope**:
- [この要件に含まないこと1]
- [この要件に含まないこと2]

---

### Requirement 2: [機能名]
...
```

## EARS形式について

EARS (Easy Approach to Requirements Syntax) は、受入基準を5つのパターンで構造化する手法:

| パターン | 構文 | 用途 |
|---|---|---|
| イベント駆動 | When [イベント], the system shall [応答] | ユーザ操作やシステムイベントへの反応 |
| 状態駆動 | While [状態], the system shall [応答] | 特定の状態が続いている間の振る舞い |
| 例外/異常系 | If [条件], the system shall [応答] | エラーや異常な入力への対処 |
| オプション | Where [条件], the system shall [応答] | 特定条件下でのみ有効な機能 |
| 普遍的 | The system shall [応答] | 常に成立すべき制約 |

すべてのパターンを各要件に網羅する必要はない。要件の性質に応じて適切なパターンを選択すること。ただし、例外系(If)は見落としやすいので意識的に検討する。

## 初期生成時のワークフロー (`/spec-requirements`)

1. **既存ドキュメントの確認**: `doc/requirements.md` が既に存在するか確認する。存在する場合は上書きではなく更新を提案する
2. **プロジェクトコンテキストの収集**: CLAUDE.md とプロジェクトの doc/ 配下を読み、現在のプロジェクト理解を把握する
3. **ペルソナの設定**: ユーザに「この要件をどのような視点からレビューしたいか」を尋ね、ペルソナを決定する（後述「ペルソナレビュー」参照）。ペルソナはプロジェクトのドメインに応じて自由に設定できる
4. **対話による要件抽出**: ユーザとの対話を通じて要件を引き出す。一度に全要件を聞き出そうとせず、1-2要件ずつ確認しながら進める
5. **Non-Goals の明確化**: プロジェクトレベルの「やらないこと」を必ずユーザに確認する。エージェント側からも拡大解釈されやすいポイントを提案する
6. **ドラフト生成と確認**: 要件定義書のドラフトを生成し、ユーザに確認を求める
7. **ペルソナレビューの実行**: ドラフト確認後、ステップ3で設定したペルソナによるレビューを実行するか確認する。実行する場合は後述のペルソナレビューフローに従う
8. **ファイル書き出し**: レビュー反映後（またはスキップ後）`doc/requirements.md` に書き出す

## 追加・修正時のワークフロー

### `/spec-requirements add`
1. 既存の `doc/requirements.md` を読む
2. 新しい要件の内容をユーザから聞く
3. 既存要件との重複・矛盾がないか確認する
4. 要件番号は既存の最大番号+1を付与する（欠番は埋めない）
5. Out of Scope を必ず含める

### `/spec-requirements modify N`
1. 既存の `doc/requirements.md` を読む
2. 指定番号の要件を表示し、修正内容をユーザから聞く
3. 修正後の要件を提示して確認を得る
4. 他の要件への影響がないか簡単にチェックする

### `/spec-requirements review`
1. 既存の `doc/requirements.md` を読む
2. 以下の観点でレビューする:
   - 各要件の Acceptance Criteria に例外系(If)が含まれているか
   - Out of Scope が十分に具体的か（曖昧な否定は効果が薄い）
   - 要件間に矛盾や重複がないか
   - Non-Goals が現在のプロジェクト状態に対して適切か
3. 発見した問題点と改善提案をユーザに報告する

## ペルソナレビュー

要件ドラフトを複数の異なるユーザ視点から検証する仕組み。Agent Teamsを活用し、各ペルソナを並列のサブエージェントとして起動する。

### ペルソナの設定

スキル起動時に、ユーザに以下を尋ねる:
- どのようなペルソナ（想定ユーザ像）を設定するか
- 何体のペルソナを用意するか（推奨: 2-4体）

ペルソナはプロジェクトのドメインに合わせて自由に設定できる。例:
- サイクリングアプリなら: ビギナー / 中級ホビー / シリアスライダー / ツーリスト
- 業務システムなら: 新人社員 / ベテラン / 管理者 / 外部監査人
- 教育アプリなら: 学生 / 教師 / 保護者

ユーザ自身が該当するペルソナも含めること。自分の視点を言語化・客観化する機会になる。

### レビューの実行

1. 各ペルソナをAgent（subagent_type: general-purpose, model: sonnet）として並列起動する
2. 各エージェントには以下を渡す:
   - ペルソナの説明（役割、経験レベル、利用シーン）
   - 要件定義書のドラフト全文
   - レビュー指示: 「このペルソナの立場から、要件の過不足・優先度の違和感・見落とされている利用シーンを指摘せよ」
3. 各エージェントの指摘を収集し、ユーザに一覧として提示する
4. ユーザが取捨選択し、反映する指摘を決定する

### レビュー結果の扱い

ペルソナからの指摘は「提案」であり、自動的に要件に反映しない。ユーザが判断して:
- 採用する指摘 → 要件に反映
- 保留する指摘 → 将来の参考として記録（任意）
- 却下する指摘 → 理由があれば Non-Goals や Out of Scope の補強に使える

却下した指摘が「やらない理由」を明確にし、否定的境界の精度を上げることもある。これはペルソナレビューの副次的な効果として重要。

## 注意事項

- 要件の記述言語は日本語。ただし EARS パターンの構文キーワード (When/While/If/Where/The system shall) は英語のまま使用する（EARS形式の可読性と一貫性のため）
- 要件番号は安定IDとして扱う。要件の削除時は欠番を許容し、番号の振り直しは行わない
- 1回の対話で完璧な要件定義を目指さない。最初は粗くても良く、`review` と `modify` で段階的に精緻化する
- 実装方法やアーキテクチャに踏み込まない。「BRouterを使って」ではなく「ルーティングエンジンにより」のように抽象化する。ただし、技術選定が要件に直接影響する場合（例: オフライン動作が必要か等）はその限りではない
