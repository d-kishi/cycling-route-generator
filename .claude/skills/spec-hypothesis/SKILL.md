# spec-hypothesis: 仮説キャンバススキル

## このスキルの目的

市谷聡啓「正しいものを正しく作る」第5章の仮説検証型アジャイル開発に基づき、仮説キャンバス(`doc/hypothesis-validation/hypothesis-canvas.yaml`)の生成・更新・ペルソナ議論・Miro描画を一気通貫で実行する。

このスキルが担うのは**「正しいものを探す」(セットベース → ポイントベース)**のプロセスである。「正しく作る」は後続の工程が担う。

## 呼び出しパターン

| コマンド | 動作 |
|---|---|
| `/spec-hypothesis` | 新規作成（起点議論 → キャンバス作成 → debate → Miro描画） |
| `/spec-hypothesis update` | 既存キャンバスの更新（トリガー確認 → 修正 → debate → Miro描画） |
| `/spec-hypothesis review` | 現状のキャンバスの整合性レビュー（変更なし、分析のみ） |
| `/spec-hypothesis validate` | 検証計画の作成（どの仮説を検証するか議論 → 検証キャンバス生成） |

## 役割分担

このスキルにおける役割は、フェーズによって異なる。

### 新規作成時（Phase 1相当）

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー（PO） | ユーザ | ビジョン・仮説の起点を提供、最終判断 |
| 開発チームリーダー | メインAgent | 技術観点での意見、仮説の構造化支援。**意見を述べる立場** |

### 議論時（Phase 2相当: debate）

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー（PO） | ユーザ | 議論への参加、最終判断 |
| スクラムマスター/ファシリテータ | メインAgent | 議論の促進・整理。**基本は中立、技術的実現性の指摘のみ例外** |
| チームメンバー | ペルソナAgent（Agent Teams） | 各ペルソナの視点から仮説を評価・議論 |

この役割切り替えは重要。新規作成時は意見を述べるが、debate時はファシリテータに徹する。

## 新規作成ワークフロー (`/spec-hypothesis`)

### Step 1: プロジェクトコンテキストの収集

CLAUDE.md と doc/ 配下を読み、プロジェクトの現状を把握する。既に `doc/hypothesis-validation/hypothesis-canvas.yaml` が存在する場合は上書きではなく `/spec-hypothesis update` を案内する。

### Step 2: ペルソナの選択

仮説の検証視点となるペルソナを選択する。

#### 前提

ペルソナの真実の源(source of truth)は `.claude/personas/persona-<id>.yaml`(`/persona` スキルが生成)。
本スキルではペルソナの新規生成・編集は行わない。ペルソナを増やしたい場合は `/persona` を起動する。

#### 手順

1. `.claude/personas/` ディレクトリを確認する
   - 1体も存在しない場合: 「ペルソナが未形成です。`/persona` で1体以上のペルソナを作成してから再実行してください」と案内し、本スキルを終了する
2. 既存ペルソナを一覧化してPOに提示する
   - 各 yaml の `id` / `name` / `description` および `attributes` の主要フィールド(動機、経験レベル、典型シナリオ、月間距離等)を抜粋表示する。全文は提示しない(コンテキスト圧迫を避けるため)
3. POに「このキャンバス議論に参加させるペルソナ」を選択させる
   - AskUserQuestion (multiSelect=true) を使用
   - **推奨**: 生成済み全ペルソナを参加候補として既定提示する(全員参加が望ましい)
   - **下限**: 2体(意味のある対立を生む最小値)
   - **上限**: 8体(opus 1M context window 環境を前提とした実用上限。Round 1+2 で各ペルソナ往復約 120K トークン × 8体 ≒ 960K で1M枠を使い切る計算)
   - 8体超のペルソナを debate に参加させたい場合は「2フェーズ方式」(コア3-4体でキャンバス作成 → 残りペルソナに意見を補完させる)を検討する
   - A/Bテストや特定視点の検証目的で意図的に少人数選抜する利用も許容する(2体以上)
   - ペルソナ数が4体を超え1画面に収まらない場合は、関連性で分けた2画面の AskUserQuestion で対応する
4. 選択結果は中間成果物 `draft-NNN.md`(Step 4で生成)と `debate-NNN.md`(Step 5で生成)に「参加ペルソナ」として記録する。本スキル側ではペルソナの専用一覧ファイルは作成しない。

#### 不足がある場合

POが「他の視点も必要だが該当ペルソナがない」と判断した場合:

- 案内: 「`/persona` で追加生成してから本スキルを再実行してください」
- 本スキルを一旦終了し、`/persona` 側にバトンを渡す

### Step 3: 仮説の起点（根源）の議論

書籍の「どこから仮説を立てるべきか？」に基づき、仮説の起点をユーザと議論する。

- 起点の種類: 目的 / 課題 / 状況 など
- メインAgentは開発チームリーダーとして技術的な観点や実現可能性の意見を出す
- 決定した起点は `root` セクションに記録し、Phase 2全体のアンカーとなる

### Step 4: 仮説キャンバスの立案

ユーザとの対話を通じて仮説キャンバスの各項目を埋めていく。

キャンバスの3区画:
- **顧客/ユーザ側** (customer): situation / overt_problem / latent_problem / alternatives
- **提供者側** (provider): purpose / vision / proposed_value / solution / advantage
- **ビジネス側** (business): channel / metrics / revenue / market_size（任意。使用するかはユーザの判断）

各仮説には以下を設定する:
- `id`: 短いID（例: C-SIT, P-VAL）
- `title`: 一言サマリー（Miroでのカード表示用）
- `hypothesis`: 仮説本文
- `confidence`: low / medium / high
- `evidence`: 根拠リスト（source + detail）。空の場合は未検証

仮説間のリレーション（`relations`）も設定する:
- `from` → `to` で方向を持つ
- `type`: supports（支持）/ derives（導出）/ conflicts（矛盾）
- `note`: リレーションの根拠

立案結果を中間成果物として `doc/hypothesis-validation/draft-NNN.md` に出力する（NNNは連番）。これにより、キャンバスがどのように構築されたかの経緯が残る。

### Step 5: ペルソナdebate

Step 2で設定したペルソナによる議論を実行する。

**判定基準: 本Step実行前に `references/debate-evaluation-criteria.md` を必ず読み、各ラウンド終了時と全体終了時に基準に従って合否判定を行う。**
判定は**メインAgent（ファシリテータ）とPOの共同**で行い、メインAgentの自己評価のみで合格としない。
不合格のまま Step 6（キャンバス更新）に進んではいけない。

判定基準の要点:

- **定量基準**: Round 1 新規アイデア3個以上/人 + 差別化1個以上、Round 2 反応2回以上/人 + 対立論点2個以上、Round 3 PO判断要求1個以上
- **失敗サイン**: F1 Round 1で収束 / F2 全員賛成 / F3 発言の交換可能性 / F4 スレッド切断 — いずれか1つでも出たら不合格
- **判定者**: メインAgentが集計・記録を出力、POが最終判断

**ペルソナAgentの役割の優先順位:**

1. **アイデア出し（最重要）**: 「自分（ペルソナ）ならこんな課題を感じる」「こういう使い方をしたい」「こんな状況がある」— 仮説の素材となる新しいアイデアを生む
2. **議論**: 他のペルソナのアイデアや既存の仮説との関係性・矛盾を議論する
3. **評価**: 既存の仮説に対する妥当性チェック（これは副次的な役割）

ペルソナAgentはレビュアーではなく、**仮説を共に作るチームメンバー**である。

**議論の進め方: 3ラウンド制**

ファシリテータ（メインAgent）が各ラウンドの区切りと移行を制御する。

#### Round 1: アイデア出し（並列）

各ペルソナAgentを並列起動（Agent Teams, subagent_type: general-purpose, model: opus）し、各自の視点から仮説のアイデアを独立に出させる。

**各ペルソナAgentに渡す情報:**
- 該当ペルソナの `.claude/personas/persona-<id>.yaml` の**全文**(attributes + checklist + broadlistening_summary を含む)
- 仮説キャンバスの全文(YAML)
- 仮説の根源(root) — これがアンカーであることを明示
- 指示: 「あなたは `.claude/personas/persona-<id>.yaml` で定義されたペルソナである。
  - `attributes` に書かれた動機・経験レベル・利用シーン・カルチャーを徹底的にロールプレイすること
  - `broadlistening_summary` の各カテゴリに含まれる一次情報・他者の体験談・自身の悩み(出典付き)を出発点として、あなた自身の課題・欲求・利用シーンに基づくアイデアを出すこと
  - 既存の仮説への意見も添えてよいが、最も重要なのは新しいアイデアの提案である
  - 根源から大幅に逸脱する提案にはその旨を明記すること」

ペルソナAgentに persona-<id>.yaml の全文を渡すのは、debateにおける**発言の交換可能性(失敗サインF3)** を構造的に防ぐため。`broadlistening_summary` の固有エピソードと出典がロールプレイの粘度を高め、ペルソナ間の差別化が言葉の表層ではなく一次情報の出所まで遡って機能する。

**Round 1終了時のファシリテータの作業:**
- 各ペルソナのアイデアを一覧化
- 論点（対立点・補完関係・共通指摘）を抽出
- Round 2に進むべき論点を特定

#### Round 2: 相互メッセージングによる議論

Agent Teamsとして各ペルソナAgentを起動し、ペルソナ同士がSendMessageで直接やり取りする。

**Round 2 着手前の準備(ファシリテータ作業):**

1. **対立軸の整理**: Round 1 のアイデア集計と論点抽出を完了する
2. **「直接の当事者+近しい立場」の組み合わせ設計**: 各対立軸ごとに「直接の当事者ペルソナ」+「近しい立場のペルソナ」を特定する。当事者だけでなく近しい立場のペルソナにも意見を求めることで、議論の二極化を防ぎ横断的視点を取り入れる
3. **知識補完オプションの確認(PO選択式)**: AskUserQuestion で以下の選択肢をPOに提示する
   - **知識補完を用意する**: 各層の固有スキル領域に紐づくテクニカル用語(他層が一次情報として知らない専門用語)で意見が出せない層に対し、他層の語彙解説を Round 2 プロンプトに含める。それでも意見が出ない層には「無し」表明を許可
   - **知識補完を用意しない**: 各ペルソナの broadlistening_summary に含まれる知識のみで議論する(知識分離の純度を優先)
   
   この選択は、ブロードリスニングによる**ペルソナの知識分離**(各ペルソナの知識境界を厳守する)と、**議論の網羅性**(全層が全論点に意見を持てる)のトレードオフ。POの判断事項とし、ファシリテータが勝手に決めない

**Round 2 の運用モデル: 監視+介入(Agent Teams 自律性とファシリテータ機能の両立):**

Agent Teams の SubAgent 間は SendMessage で双方向に直接対話できる。各ペルソナがラリー結果を毎回 main にも CC すると、main コンテキストを浪費し Agent Teams の自律性も生かしきれない。一方で main がファシリテータ役割(脱線是正・収束促進・技術指摘)を担うには、議論の流れを把握しておく必要がある。両立のため以下のモデルを採用する:

1. **議論プロセスは sub 間ラリーで完結させる**
   - ペルソナ間の対立・合意・連合形成は SendMessage で直接やり取り
   - ラリー全文を main に CC しない(冗長)

2. **main は peer DM visibility で監視する**
   - 各 sub の idle_notification には `summary: "[to persona-X] <短い概要>"` が含まれる(Agent Teams の peer DM visibility 機能)
   - main はこの summary を継続監視し、議論の流れを把握

3. **必要時のみ main が inbox JSON を pull して全文確認**
   - inbox path: `~/.claude/teams/<team-name>/inboxes/<persona-id>.json`
   - 取得方法: Bash で `cat` または Python で JSON 解析(各メッセージの `from` / `text` / `timestamp` を読む)
   - summary では判断できない時(脱線疑い・複雑な合意形成・新規論点判定)に全文確認

4. **main の介入トリガー(SendMessage で sub に直接介入する)**:
   - **脱線**: 根源(root)から大幅に逸脱する議論に発展した時
   - **収束**: 対立論点が合意に向かい次論点に移れる時
   - **技術指摘**: ペルソナの提案が技術的に実現困難な時(基盤プラットフォーム制約・データ可用性・実装コスト等)
   - **新規論点提示**: ラリー中に発見されなかった重要論点を投入する時
   - **中止判断**: 議論が空回り(F4スレッド切断兆候)している時に当該ペアのラリーを停止

**各ペルソナAgentへの指示(Round 2 開始時):**
- 「Round 1で他のペルソナから出たアイデアと論点を共有する。あなたのペルソナの立場から、これらのアイデアに対する反応(同意・反論・補足・新たな視点)を SendMessage で他のペルソナに直接伝えてほしい」
- **各論点について「当事者性: 直接の当事者 / 近しい立場 / 関係薄い」を明示**してから意見を述べること
- 直接の当事者: 自分の層の固有課題 → 主張を強化
- 近しい立場: 自分の経験から見える横断的視点 → 統合提案や橋渡し
- 関係薄い: 「外野の所感」レベルでも歓迎(層代弁者の視点として価値あり)
- 知識補完オプション「用意する」を選択した場合: 専門用語で意見が出せない層は「**知識補完を受けた上での意見無し**: <理由>」と明記してOK(評価上減点しない)
- 知識補完オプション「用意しない」を選択した場合: 自身の broadlistening_summary の範囲で意見を述べ、範囲外なら「(自身の知識範囲外)」と明記
- 特に対立点については、なぜそう考えるかの根拠を示すこと

**sub→main の流量制御(各ペルソナへの指示に含める):**
- ラリー全文を main(team-lead) に毎回送らない(冗長、main コンテキスト浪費)
- 以下の場合のみ main に短い alert を送る:
  - 重要な合意形成があった時(連合確定、論点クローズ等)
  - 新規論点を提起した時(既存10論点に含まれない発見)
  - 脱線懸念がある時(根源から逸脱しそう、または他ペルソナの議論が空回り)
- 最終意見はラウンド終了時に1回 main に送る(ラウンド全体の自分の立場サマリ)
- main は peer DM visibility(idle_notification の summary)で議論を継続監視している前提で、フル報告は不要

**SendMessage の宛先について:**
- Round 1 で起動したペルソナAgentは Round 2 でも継続利用する(persona-yaml 全文の再送が不要)
- SendMessage の宛先は **agent_id** で指定する(エージェント名では送信失敗するケースがある)
- agent_id は Round 1 起動時の Agent ツールレスポンスに含まれる

**Round 2終了時のファシリテータの作業:**
- 各 sub の inbox JSON を pull して全ラリーログを取得
  - 例: `cat ~/.claude/teams/<team-name>/inboxes/persona-<id>.json` または Python で JSON 解析
  - inbox には `from` / `text` / `summary` / `timestamp` フィールドを持つメッセージ配列が時系列で蓄積されている
- 議論の経過と結論を整理して debate-NNN.md に集約
- 合意が形成された点と、対立が残った点を明確化
- 介入実績(脱線是正、収束促進、技術指摘、新論点提示)も記録(debate品質メタ評価素材になる)

#### Round 3: POの参加と収束

ファシリテータがRound 1-2の全体を整理してPOに提示する。

- 各ペルソナのアイデア一覧
- 議論で合意が形成された点
- 対立が残っている論点（各立場の根拠付き）
- ファシリテータとしての技術的補足（あれば）

POが以下を判断する:
- 各論点に対する採否の決定
- 追加の議論が必要かどうか

**debate結果の記録:**
- Round 1〜3の全経過を `doc/hypothesis-validation/debate-NNN.md` に出力する（NNNは連番）

### Step 6: キャンバス更新

debate Round 3でのPO判断を踏まえ、キャンバスを更新する。

- 採用されたアイデア → キャンバスの仮説・リレーションに反映
- 保留されたアイデア → debate記録に記載（任意）
- 却下されたアイデア → 理由があれば仮説の否定的境界として活用可能

キャンバス更新後、YAMLファイル (`doc/hypothesis-validation/hypothesis-canvas.yaml`) に書き出す。

### Step 7: Miro描画の確認と実行

**重要: Miroはバージョン管理が独立しているため、描画前に必ずPOの承認を得る。**

1. 更新後のキャンバスの概要をPOに提示する
2. 「Miroボードを更新しますか？」と確認する
3. 承認された場合のみ、`references/miro-canvas-rendering.md` の手順に従い `diagram_create` で描画する
4. キャンセルされた場合、YAMLファイルのみ保存してスキルを終了する

描画手順・DSL構造・MCP制約・手動調整前提の運用は `references/miro-canvas-rendering.md` を参照する。要点のみ抜粋すると、原典の14区画レイアウトをflowchart cluster で表現し、relations を connector で描画する。生成後の位置調整はユーザがMiro UIで手動で行う前提。

## 更新ワークフロー (`/spec-hypothesis update`)

### Step 1: 既存データの読み込み

以下を読み込む:
- `doc/hypothesis-validation/hypothesis-canvas.yaml` — 既存キャンバス。存在しない場合は `/spec-hypothesis` での新規作成を案内する
- `.claude/personas/persona-*.yaml` — 現在利用可能なペルソナ群 (真実の源)
- `doc/hypothesis-validation/debate-NNN.md` の最新ファイル — 前回どのペルソナで議論したかを把握する。複数ある場合は最大番号のものを採用

### Step 2: 更新トリガーの確認

ユーザに更新のトリガー（理由）を尋ねる。トリガーによってdebateの注力点が変わる:

| トリガー | debate時の注力点 |
|---|---|
| インタビュー結果の追加 | 新しいevidenceが既存仮説を支持するか否定するか。confidenceの更新 |
| PO意思による仮説修正 | 根源（root）からの逸脱がないか確認。逸脱する場合は根源自体の見直しを検討 |
| 新ペルソナの追加 | 新しい視点からの既存仮説の再評価。見落とされていた課題やユースケースの発見 |
| ユーザフィードバック（将来） | 運用データに基づく仮説の検証度（confidence）更新。仮説の棄却判断 |

### Step 3: キャンバスの修正

トリガーに応じてキャンバスを修正する。

- インタビュー結果追加: 該当するhypothesisのevidenceリストに追加。confidenceの見直し
- PO意思による修正: ユーザとの対話で修正内容を確認
- ペルソナ追加: 追加したいペルソナが `.claude/personas/` に未生成の場合、`/persona` で先に生成してから本スキルに戻る。既存ペルソナを debate 参加メンバーに加えるだけなら、Step 4(debate)実行前にメンバー再選択するだけでよい(本スキル側で専用一覧ファイルは持たない)

### Step 4: ペルソナdebate

新規作成時のStep 5と同じフロー。ただし以下の差分に注意:

- 更新トリガーを各ペルソナAgentに伝え、そのトリガーに関連する仮説を重点的に議論させる
- 前回のdebateで既に合意済みの論点を再度蒸し返さないよう、変更点にフォーカスする指示を含める

### Step 5-7: 議論結果確認 → キャンバス更新 → Miro描画

新規作成時のStep 6-7と同じフロー。

## レビューワークフロー (`/spec-hypothesis review`)

1. `doc/hypothesis-validation/hypothesis-canvas.yaml` を読み込む
2. 以下の観点でレビューする:
   - **仮説の一本線の検証（最重要）**: 状況 → 課題 → 提案価値 → 実現手段 が `relations` を通じて論理的に一貫して繋がっているか。根源（root）から最終的な実現手段まで、途切れなく辿れる一本の線が存在するか確認する。途切れている箇所や論理の飛躍がある箇所を特定する
   - 根源（root）と各仮説の整合性: 根源から論理的に導出できるか
   - confidence と evidence の整合性: evidence があるのに confidence が low のまま等
   - relations の妥当性: 欠けているリレーションはないか、不要なリレーションが混在していないか
   - 未検証の仮説（evidence が空）の特定: 一本線上にある未検証の仮説は特に優先度が高い
   - conflicts（矛盾）の有無: 矛盾するリレーションが放置されていないか
3. 発見した問題点と改善提案をPOに報告する（キャンバスは変更しない）

## 検証計画ワークフロー (`/spec-hypothesis validate`)

仮説キャンバスの中から「何を検証すべきか」を議論し、次工程（インタビュー、モックアップ作成等）の起点となる検証キャンバスを生成する。

### このワークフローの責務境界

- **このスキルが担う**: どの仮説を検証に持っていくかの議論・決定、検証キャンバスの生成
- **このスキルが担わない**: インタビュー設計・実施、モックアップ作成、検証結果の分析（これらは別工程）

### Step 1: 既存データの読み込み

- `doc/hypothesis-validation/hypothesis-canvas.yaml` — 現在のキャンバス
- `.claude/personas/persona-*.yaml` — 現在利用可能なペルソナ群

### Step 2: 検証対象の仮説を議論

POと対話し、以下の優先度基準で検証すべき仮説を特定する:

1. **一本線上にある未検証仮説**: 仮説の一本線が途切れるリスクが最も高い。特にconfidenceがlowの仮説
2. **根源に近い仮説**: 根源に近い仮説が崩れると、そこから派生する全ての仮説が崩れる
3. **conflictsが存在する仮説**: 矛盾が解消されていない箇所は検証が急務
4. **evidenceが空の仮説**: まだ何の根拠もない仮説

### Step 3: 検証キャンバスの生成

検証対象として選定した仮説ごとに、以下を整理して `doc/hypothesis-validation/validation-canvas-NNN.yaml` に出力する:

```yaml
validation:
  - hypothesis_id: <検証対象の仮説ID>
    hypothesis_title: <仮説タイトル>
    current_confidence: <low / medium / high>
    validation_goal: <この検証で何を明らかにしたいか(対象セグメントに広く共有されているか等)>
    suggested_methods:
      - "対象ユーザへのヒアリング"
      - "ペルソナAgentへのインタビュー"
      - "SNS/コミュニティでのニーズ調査"
    success_criteria: <何をもって仮説が支持/棄却されたと判断するか(例: N名中M名以上が同様の課題感を表明)>
    priority: <high / medium / low>
```

- `validation_goal`: この検証で何を明らかにしたいか
- `suggested_methods`: 検証方法の候補（実際にどの方法を取るかは次工程が決定）
- `success_criteria`: 何をもって仮説が支持/棄却されたと判断するか
- `priority`: high / medium / low

### Step 4: POの確認

検証キャンバスの内容をPOに提示し、承認を得る。必要に応じて修正する。

## 仮説の根源（アンカー）による制約

Phase 2（debate）での議論が発散しないよう、以下の制約を設ける:

- 全ての議論は根源（root）に照らして妥当性を判断する
- 根源から大幅に逸脱する提案にはフラグを立てる
- 逸脱が合理的である場合は、根源自体の見直しをPOに提案する（根源の変更は重大な判断であることを明示する）

この制約は、AIモデルが無限の情報にアクセスできるがゆえに議論が収束しにくい問題への構造的対策である。

## 注意事項

- 仮説キャンバスの記述言語は日本語
- 仮説番号（id）は安定IDとして扱う。削除時は欠番を許容する
- 1回の対話で完璧なキャンバスを目指さない。`update` と `review` で段階的に精緻化する
- 実装方法やアーキテクチャの詳細に踏み込まない。このスキルは「何を作るか」の探索に集中する
- debate時、メインAgentはファシリテータとして中立を保つ。技術的実現性に関する指摘のみ例外として意見を述べてよい
