# 仮説キャンバスのMiro描画手順

## 設計方針

仮説キャンバスをMiroで可視化する際は、以下の二点を両立させる:

1. **原典(市谷聡啓「正しいものを正しく作る」)の14区画レイアウト構造**を表現
2. **仮説間の関連性(relations)** をconnectorで可視化

両立のために `mcp__miro__diagram_create` の **flowchart形式 + cluster + connector** を使用する。

## 14区画と仮説IDの対応

| 原典区画 | 役割 | 該当する仮説ID(典型的な命名例) |
|---------|------|----------------------------------|
| 目的 | われわれはなぜこの事業をやるのか | P-PUR |
| ビジョン | 中長期的に顧客にどういう状況になってもらいたいか | P-VIS |
| 状況 | どのような状況にある顧客が対象か | C-SIT |
| 傾向 | 同じ状況にある人が一致して行うこと | (任意) |
| 顕在課題 | 顧客が気づいている課題 | C-OVR系 |
| 潜在課題 | 多くの顧客が気づいていない課題 | C-LAT |
| 代替手段 | 課題を解決するために顧客が現在取っている手段 | C-ALT |
| チャネル | 状況にある顧客たちに出会うための手段 | (任意) |
| 提案価値 | われわれは顧客をどう解決状態にするのか | P-VAL |
| 優位性 | 提案価値の実現や提供に貢献するリソース(資産) | P-ADV |
| 評価指標 | どうしてこの事業が進捗していると判断できるか | (任意) |
| 実現手段 | 提案価値を実現するために必要な手段 | P-SOL |
| 収益モデル | どうやって儲けるのか | P-MOD |
| 市場規模 | 対象となる市場の規模感 | (任意) |

YAMLに該当する仮説が無い区画は、空にせず「(未定義)」ダミーノードを置いて区画自体は表示する。これにより「v1で未策定の領域」が可視化される。

戦略系仮説(P-MVP等、原典に対応区画が無いもの)は **実現手段 cluster に併置** する(MVPは実現手段の段階分けと解釈できるため)。

## flowchart DSL構造

```
graphdir TB
palette #fff6b6 #adf0c7 #c6dcff #e7e7e7

# === ノード定義 ===
# n# <ラベル> <flowchart-data|process|decision|terminator> <color_index>
n1 C-SIT: 状況の要約 flowchart-data 0
n2 C-OVR (層名): 課題タイトル flowchart-process 0
...
n16 (未定義) flowchart-data 3   # 空区画用ダミー

# === コネクタ定義 ===
# c <source> <ラベル|-> <target>
c n1 中級 n2
c n2 - n7
c n7 目的導出 n9
...

# === クラスター定義(必ず最後) ===
# cluster c# "<区画名>" <node_id1> <node_id2> ...
cluster c1 "目的" n9
cluster c5 "顕在課題" n2 n3 n4 n5 n6
...
```

### ノードオブジェクトの使い分け

| object | 用途 |
|--------|------|
| flowchart-terminator | 目的・ビジョン(角丸、開始/終了の意味合い) |
| flowchart-process | 課題・提案価値・実現手段など主体的な仮説 |
| flowchart-data | 状況・傾向(平行四辺形、入力/参照データ的位置づけ)、ダミーノード |
| flowchart-decision | 判断分岐がある場合(本キャンバスでは通常未使用) |

### 色の運用(palette index)

| index | 色 | 用途 |
|-------|----|----|
| 0 | #fff6b6 (薄黄) | 顧客側仮説(C-*) |
| 1 | #adf0c7 (薄緑) | 提供者側仮説(P-PUR/VIS/VAL/SOL/ADV) |
| 2 | #c6dcff (薄青) | 戦略系仮説(P-MVP/P-MOD) |
| 3 | #e7e7e7 (灰) | 「(未定義)」ダミーノード |

### コネクタラベル

YAMLの `relations` の `note` を要約してconnectorラベルにする。長文は短縮:

- `中級層の状況がマンネリ課題を生む` → `中級`
- `潜在課題から目的が導出される` → `目的導出`
- `代替手段では一元化できないことが提案価値の差別化根拠` → `差別化根拠`

ラベルが思いつかない/接続関係が自明な場合は `-` で空にする(線のみ表示)。

## MCP制約と運用

### 自動レイアウトであること

`diagram_create` のflowchart形式は **Miroの自動レイアウトエンジン** が cluster間の配置を決定する。DSL からは座標指定不可で、`graphdir <方向>` (TB/LR/BT/RL) のみ全体方向を指定できる。

そのため、原典の固定グリッド(上端2区画/中央6カラム/下端2区画)を **完全再現することはできない**。Miroは connector の流れを優先してレイアウトする。

### 生成後の修正

| 操作 | 可否 |
|------|------|
| MCPツールから cluster/node の位置を変える | **不可** (`layout_update` の対象外、stencil_shape は更新できない) |
| Miro UI上でユーザが手動で配置を変える | 可能(通常のMiroオブジェクトとして編集できる) |
| ノードのテキスト編集 | UI上で可能 |
| connector の付け替え/ラベル編集 | UI上で可能 |
| ノード/cluster の追加・削除 | UI上で可能 |

**運用上の前提**: 自動レイアウトの結果が原典通りでなくても許容し、最終的な位置調整はユーザがMiro UIで手動で行う。AIエージェント側はあくまで「論理構造(ノード+関連線+cluster所属)を正確に出力する」ことに責任を持つ。

### diagram削除

diagram自体を削除するMCPツールは現状提供されていない。試行錯誤で再生成すると古いdiagramが残るため、**生成は1発で済ませる**ことを目指す。やむを得ず再生成する場合は、古いdiagramの手動削除をユーザに依頼する。

## 描画手順

1. **YAMLから抽出**:
   - `canvas` 配下の各仮説 → ノード化(id/title/区画分類)
   - `relations` → connector化(from/to/note)
2. **14区画cluster定義**: 該当仮説が無い区画には「(未定義)」ダミーノードを1つ配置
3. **DSL組み立て**: 上記構造に従い、ノード→コネクタ→クラスターの順で記述(クラスターは必ず最後)
4. **POへの確認**: 描画前に「Miroボードを更新しますか?」と必ず確認(Miroはバージョン管理が独立しているため上書きリスクあり)
5. **`diagram_create` 実行**: 既存ボードがあれば `miro_url` を指定、無ければ新規作成
6. **配置位置**: 既存diagramがある場合は、十分な余白(200px以上)を取って下方/右方に配置
7. **完了報告**: `moveToWidget` 付きURLをユーザに提示。手動調整が必要な旨を伝える

## ボードURL管理

- 描画対象のボードURLはYAMLの `meta.miro_board_url` に記録し、以降の実行で再利用する
- 初回はユーザに尋ねる。新規ボード作成は **必ずユーザ確認を取る**(`board_create` 実行前)

## 注記: 描画形式の選択肢

`layout_create` 系(FRAME/STICKY/SHAPE/TEXT)で原典レイアウトを精密に表現することは可能だが、現MCPツールセットでは **connector(線)を描く手段がそちらの系統には無い**。よって関連線を表示したい場合は flowchart 系一択となる。
- 関連線を諦めて精密レイアウトを取る → `layout_create` でFRAME+TEXT
- 関連線を取って自動レイアウトを許容する → `diagram_create` でflowchart (本ドキュメント推奨)

仮説キャンバスは「仮説の一本線」を可視化することが核心目的のため、後者を採用する。
