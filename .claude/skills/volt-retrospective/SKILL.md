# volt-retrospective: ボルト振り返りスキル

## このスキルの目的

ボルト(機能単位の完了サイクル、時間〜日単位)の終了時に、観察された改善候補・運用フィードバック・不具合等を構造的にハーネス(skill / rule / memory / docs)へ反映する。

session-kickoff (開始時 skill) と対称な「ボルト終了時 skill」として、開発サイクルの開始と終了を skill レベルで対称化する。

## 役割

このスキルが担うのは **「ボルトで得た知見を体系的に蓄積するプロセス」** である。具体的な改善内容(何をどう変えるか)は、振り返り素材とプロジェクトの方針に従って判断する。

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー(PO) | ユーザ | 改善の優先度判断、「やりすぎ」抑制の最終判断 |
| 振り返り進行役 | メインAgent | 素材収集 → 分類 → 反映 → 記録 → commit の進行、判断の透明化 |

## なぜこのスキルが必要か

振り返りを暗黙知のまま進めると以下の問題が発生する:

1. **再現性低下**: 進行役 (メインAgent) が変わるたびに手順が再発明され、知見の蓄積効率が落ちる
2. **反映漏れ**: 素材を口頭で消化するだけで終わり、skill / rule / memory に定着しない
3. **やりすぎ**: 「1ボルト1-2改善ルール」を意識せず、複数改善を同時に積んでハーネスを肥大化させる
4. **commit粒度の混乱**: 関連改善と独立改善を区別せずまとめてしまい、後で履歴を辿れない

本スキルは上記4つを構造的に防ぐ。

## 手順

### Step 1: 振り返り素材の収集

以下の場所から振り返り素材(改善候補・運用観察・不具合・新規提案等)を収集する:

- **auto memory の next-session-plan 系メモリ**: 「振り返り素材」「次回振り返りで扱う」等のセクションに蓄積されているもの
- **直近のセッション履歴**: 本セッション内で観察されたが未消化のフィードバック
- **直近の commit log と差分**: 実装した内容と、その過程で発生した問題・改善余地
- **直近の作業ログ / 議事録**: ボルト中に作成された設計書・議論記録から「次回扱う」と記録された項目
- **失敗・エラー履歴**: 同じミスを2回以上繰り返した記録(リカレント問題)

**素材0件の場合**: 「振り返り対象なし」と報告して本スキルを終了する。無理に振り返りを生成しない。

### Step 2: 素材の分類

収集した素材を以下の2軸で分類する。

#### 軸1: 反映先

| 反映先 | 該当する素材 |
|---|---|
| **skill本体** (`.claude/skills/*/SKILL.md`) | skill のワークフロー手順自体の改善、起動条件の明確化、Step 追加 |
| **skill references** (`.claude/skills/*/references/*.md`) | 詳細手順、判定基準、テンプレート、技術制約等 |
| **プロジェクト rule** (`.claude/rules/*.md`) | 開発全般で常時守るべきルール、表記規則、運用方針 |
| **グローバル rule** (`~/.claude/rules/*.md`) | プロジェクト横断のルール(本プロジェクト以外でも適用したいもの) |
| **memory** (`~/.claude/projects/<...>/memory/*.md`) | 経緯記録 (feedback)、外部資源ポインタ (reference)、進行中の状態 (project)、ユーザ情報 (user) |
| **プロジェクト docs** (CLAUDE.md / docs/*) | プロジェクト全体方針、技術スタック、ハーネス設計思想 |

#### 軸2: 緊急度

| 緊急度 | 判定基準 |
|---|---|
| 高 | 次ボルトに即必要(これがないと次の作業が破綻する) |
| 中 | 数ボルト以内に消化したい(放置すると同じ問題が再発する) |
| 低 | 経験則として記録しておく(将来活かせる知見、頻度低い) |

#### 分類のコツ

- **同じ素材が複数の反映先候補を持つ場合**、最も恒久性の高いものを主とし、他は補助参照(memory に「Rule化済み」フラグを残す等)
- **「やった瞬間に効くもの」は skill / rule、「経緯として残すもの」は memory** が原則
- **誰がどう使うか**を想像して反映先を決める(Claude が常時参照するなら rule、起動時に参照するなら skill)

### Step 3: 「1ボルト1-2改善」ルールの適用判断

複数項目を一度に反映する場合、「やりすぎ」リスクを判断する。

#### 集約 vs 分割の判断

- **関連性の高い小改善の集まり**(例: 同一 skill への複数箇所修正、互いに参照し合う rule + memory) → 1個と数えてOK、まとめて消化
- **独立した複数改善** → 別ボルトに分散することを推奨。今ボルトでは1-2個に絞る
- **判断に迷う場合は PO に相談**: 「以下N件を消化候補としていますが、今ボルトでまとめるか / 次ボルトに分散するか」を AskUserQuestion で確認

#### 「やりすぎ」シグナル

以下が出たら立ち止まって PO に確認:

- 反映ファイルが5個以上に及ぶ
- 反映先が3カテゴリ以上(skill + rule + memory + docs 等)にまたがる
- ハーネス全体の構造を変えるレベルの修正(skill 新設、CLAUDE.md 大幅改訂等)

### Step 4: 反映実施

決定した改善を該当ファイルに反映する。

#### 必ず守る原則

- **skill / rule / docs は plugin 配布対応の汎用記述で書く** (グローバルルール `skills-design-principle` を遵守。特定ワークフロー / プロジェクト依存記述を避ける)
- **識別子・略号は単独で書かない** (プロジェクト rule `notation` を遵守。「識別子+一言概略」セットで表記)
- **日本語テキスト出力時は japanese-sentence-break skill を適用** (句点後改行、日本語/半角英数の境界スペース処理等)
- **ユーザが明示した個別ルール** (CLAUDE.md / グローバル rule / プロジェクト rule) を全て遵守

#### 反映の順序

1. 影響が小さく独立性の高いものから先に反映 (skill本体 → references → rule → memory の順が一例)
2. 関連性の高い改善はまとめて Edit
3. 反映完了後、各ファイルで grep / 簡易レビューして「他に汎用化漏れがないか」をチェック

### Step 5: memory 更新

振り返り素材の所在 memory (例: `project_next_session_plan.md` 等) を更新する。

- 消化済み項目に「✓完了」フラグと日付を付ける
- 新規 memory が必要な場合は追加し、`MEMORY.md` のインデックスにエントリを追加
- 反映先に新規ファイルを作った場合(新 rule / 新 skill 等)、関連 memory から参照リンクを張る

### Step 6: 完了報告とユーザ確認

反映と memory 更新まで終えた段階で、commit に進む前に**ユーザに確認**する。以下を簡潔に提示する:

- 消化した素材の件数と内訳(反映先別)
- 残った素材があれば次回振り返りに持ち越す旨と、所在 memory の更新状況
- 反映ファイルの一覧(これから commit 対象となるもの)

そのうえで「commit & push に進んで良いか」をユーザに確認する。理由:

- 振り返りの反映内容は **ハーネスの将来挙動を変える** ものが多く、commit 前に PO の最終確認を経る価値がある
- ユーザが「内容は OK だが commit メッセージの粒度を変えたい」「特定ファイルだけ別ボルトに回したい」等の微調整を入れる余地を残す
- commit & push は実質的に元に戻しにくい操作のため、確認ステップを挟む

ユーザ承認が得られたら Step 7 へ進む。修正指示があれば該当箇所に戻って再反映してから本ステップに戻る。

### Step 7: commit & push

ユーザ承認を得たうえで commit & push を実施する。

#### コミット粒度

- **1コミット1意図** (Conventional Commits の精神に従う)
- 関連改善 (例: 同一 skill への複数箇所修正) は1コミットにまとめてOK
- 性質の異なる改善 (例: 設計検証成果物 + ハーネス改善 rule 追加) は別コミットに分割

#### コミットメッセージ

- Conventional Commits 形式 (feat / fix / chore / docs / refactor / test) のうち適切な type を選ぶ
- HEREDOC 形式で本文に「何を、なぜ」を記載
- プロジェクト規約に沿った Co-Authored-By フッタを付ける(該当があれば)

#### main直 vs feature/* の判断

プロジェクトの分岐ルール (CLAUDE.md / 関連 memory / `.claude/rules/git-workflow.md` 等) に従う。判断に迷う場合は PO に確認。

#### push後の最終提示

commit ハッシュ一覧をユーザに提示して本スキルを終了する。

## 振り返りで扱わないもの

以下は本スキルの対象外:

- **新機能の実装** — 通常のボルト実行で扱う
- **不具合の即時修正** — 振り返り素材化せず、その場で修正(振り返りは「再発防止策の蓄積」として扱う)
- **未確定の議論** — PO の意思決定が必要な未確定論点は、振り返り素材ではなく別途議題として扱う

## 関連スキル / ルール

- `session-kickoff`: 対称的な開始時 skill。本スキルとセットでボルトの開始と終了を構造化する
- グローバル rule `skills-design-principle`: skill / rule / docs を汎用記述に保つ責務(本スキル Step 4 で遵守)
- プロジェクト rule `notation`: 識別子・略号の表記ルール(本スキル Step 4 で遵守)
- グローバル rule `communication`: AskUserQuestion 積極活用 (本スキル Step 3 の集約 / 分割確認で活用)

## 注意事項

- 本スキル自体も汎用記述で書く(plugin 配布想定)。特定プロジェクトの skill 名・ファイル名・ドメイン語彙を埋め込まない
- ユーザが「最小構成主義」「やりすぎ抑制」志向の場合、Step 3 の集約 / 分割判断を慎重に行う
- 振り返り素材の所在 memory がプロジェクトごとに異なる場合、そのプロジェクトの慣習に従ってどこを読むかを判断する
