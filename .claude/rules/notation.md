# 識別子・略号の表記ルール

## ルール本体

識別子・ID・略号(`D1` / `C-OVR-SUP` / `(1)` / `Q1` / `Volt 6+` / `MVP` / `BRM` 等)を**単独で書かない**。必ず「識別子+一言概略(1〜2語、目安10〜20文字)」のセットで表記する。

これは仮説検証段階だけでなく、**開発全般・全ての文脈**に適用する常時ルール。

## なぜこのルールが必要か (Why)

ユーザは識別子だけ提示されても、その内容を理解するために逐一ドキュメント全体を解読する必要がある。これは認知負荷が高く、対話・レビューの効率を著しく落とす。

特に以下の場面で問題が顕在化する:

- AskUserQuestion の選択肢説明で `Vol 6+ で 7軸の (1)-(3) を実装` のように略号が連鎖すると、選択肢の意味判断ができない
- レビュー依頼で `D5 を採用しました` と書かれても、D5 が何の論点かを思い出すコストが高い
- セッションを跨いだ memory 参照で、過去セッションの識別子を当時の文脈なしに読まされる

## 表記の具体例

### NG (識別子単独)

- 「D1 を全層共通に拡張します」
- 「C-OVR-SUP を再定義します」
- 「Vol 6+ で(1)-(3)を実装」
- 「Q1-Q4 を確定」
- 「MVPはhobby中心」(MVPが文中で初出時)

### OK (識別子+一言概略)

- 「D1 (対象範囲拡張) を全層共通に拡張します」
- 「C-OVR-SUP (ロング志向の7軸統合計画) を再定義します」
- 「Volt 6+ (将来サブスクモジュール段階) で7軸統合計画のうちコア軸の(1)PC通過時刻判定・(2)補給ポイント時系列マッピング・(3)仮眠予算配分を実装」
- 「Q1-Q4 (対象範囲・信号情報・anchor・拡張順序) を確定」
- 「MVP (最初リリース版) はhobby中心」

## 適用範囲

- ユーザ宛のテキスト出力(回答・要約・選択肢説明)
- AskUserQuestion の question / options.label / options.description
- ドキュメント記述(md/yaml のコメント・hypothesis 文・hypothesis のタイトル等)
- memory ファイル
- コミットメッセージ・PR 説明

**例外**: 表形式で識別子列が独立している場合(`| ID | 説明 |` のような構造)、ID列は識別子単独で可。説明列で内容を補う。

## 略号一般への適用

識別子だけでなく、**略号全般**(専門用語の頭字語、独自略語)にも適用する:

- `MVP` → `MVP (最初リリース版)`(初出時)
- `BRM` → `BRM (Audax Japan認定ブルベ)`(初出時)
- `OAuth` → `OAuth (外部サービス認証連携)`(初出時)
- `TSS/IF/CTL/ATL` → `TSS/IF/CTL/ATL (TrainingPeaksの負荷指標群)`(初出時)

同一ドキュメント・同一会話ターン内で複数回出る場合、初出時のみ展開し、2回目以降は識別子のみで可。

## 違反した場合

- ユーザから「分かりません」「説明して」と指摘されたら即座に該当箇所を一言概略付きで再表記する
- 同一セッションで2回以上違反したら、原因(無意識の略号化 or テンプレ流用)を内省し、以後の出力で警戒を強める
- 違反が複数セッションをまたいで観察される場合、本ルール自体の表現が不十分と判断し、具体例追加・適用範囲明確化等の改訂を検討する
