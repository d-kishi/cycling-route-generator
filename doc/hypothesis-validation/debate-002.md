## 修正案ドラフト3件 (PO起点)

### 修正案 A: 新規仮説 C-OVR-TIME を追加

```yaml
id: C-OVR-TIME
title: "所要時間見積もりギャップ(中級以上共通)"
hypothesis: |
  中級以上のサイクリスト(hobby/serious/endurance)は「実走時間=移動時間+補給/休憩」の所要時間を事前に見積もろうとするが、既存ツールは平均速度の単純割り算しか提供せず、補給・休憩・信号停止・上り下りでの速度変化を含む実態と乖離する。出発時刻・帰宅時刻・補給タイミング・本番ペース配分の見積もりに失敗し、ライド体験の満足度が下がる。
confidence: low
```

既存 C-OVR-SUP は endurance 固有のロング行程計画として残し、所要時間見積もりは層横断課題として独立させる。

### 修正案 B: P-VAL D5 粒度階層を再定義

修正前: serious=フル数値、endurance=行程指標、hobby=軽量、beginner=3択、casual=構造化された気分

修正後: serious=フル数値(HRV/CTL/ATL)+本番ペース配分、endurance=長時間行程指標(グロス速度・補給間隔・仮眠予算)、**hobby=軽量グロス速度(出発/帰宅時刻ラフ表示+補給間隔目安)**、**serious(練習時)=Z2 Base時間配分(グロス速度の練習版)**、beginner=3択、casual=構造化された気分

### 修正案 C: P-MVP Volt 1 に層横断機能を先取り組み込み

修正前 Volt 6+: endurance層フルグロス速度シミュ、serious層HRV/CTL/ATL連動は将来サブスクモジュール

修正後: **Volt 1 (MVP) に追加**: 平均速度+補給/休憩想定時間ベースの所要時間ラフ表示(層横断機能、軽量版グロス速度)
Volt 6+: endurance層フルグロス速度シミュ、serious層HRV/CTL/ATL連動は将来サブスクモジュール (既存通り)

## Round 1 各ペルソナのアイデア(集計)

### persona-hobby (中級ホビーロード) — 3個提案

1. **都内発80-120kmでの「帰宅時刻ズレ問題」は中級ホビーの隠れペイン**
   - 当事者性: 直接の当事者
   - 内容: 平均20km/h×時間の頭計算でも信号・集合写真・ペース乱れで毎回30-90分ズレる。家族持ち中級層にとって「帰宅時刻が読めない」は地味だが継続障壁。「グロス速度」と呼ばないだけで実態は皆やっている
   - 修正案A強く支持(C-OVR-SUPと混ぜると粒度が薄まる)
   - 出典: cyclinglabs.net「シンプルツールが求められる」、PistonHeads UK 2024 Correvor氏

2. **修正案B「軽量グロス速度」の具体化 — 数字を見せすぎない実装**
   - 当事者性: 直接の当事者
   - 内容: 数値そのものではなく「12:00昼飯、16:30帰宅予定」の時刻表的要約。ルート上に「☕補給 12:10」「🍜昼食 13:00」「🏠帰宅 16:30(±30min)」のピン表示。広い誤差幅明示が「数値駆動になりすぎない」hobby層的ちょうどよさ
   - 修正案B支持、ただし表現の具体化が必要
   - 出典: cyclinglabs.net「シンプルツールが求められる」、Komoot App Store 2025/11 ★2「訳の分からない遠回り」、debate-001 D5「hobby=軽量」

3. **修正案C(MVP組み込み)はホームショップ起点フローと組み合わせて欲しい**
   - 当事者性: 直接の当事者
   - 内容: 「9:00ショップ集合 → 80km → 14:30ショップ戻り(休憩30分込) → 機材点検後解散」という時刻ベース計画は、ホームスポット抽象化とグロス速度を組み合わせて初めて成立。MVP単体では中途半端
   - 修正案C支持、ただし P-VAL要素5(ホームスポット抽象化)との連動をMVP段階で明示
   - 出典: persona-hobby maintenance_practice、debate-001 persona-hobby Round 1

### persona-beginner (ビギナー) — 4個提案

1. **「あと何分で家に帰れるか」が初心者にも切実**
   - 当事者性: 直接の当事者
   - 内容: Googleマップが「30km・1時間40分」と出すが実際は信号・休憩・写真でいつも倍近くかかる。家族に「何時に帰る」と言って2時間遅れる経験が、河川敷を一周するだけの安全圏に閉じこもる原因
   - 修正案C強く支持(MVPに入れないと初心者層の入口体験が成立しない)
   - 出典: persona-beginner navigation_tool、road55.com「補給は30分〜1時間に1回」、persona-beginner typical_scenario

2. **「補給・休憩を含むべき」という発想自体が初心者には抜けている**
   - 当事者性: 近しい立場
   - 内容: 「自分の脚力不足で予定より遅れた」と恥じる心理。「補給・休憩は予定外じゃなく標準装備です」というメッセージを画面で明示すべき
   - 修正案A支持・拡張(初心者UIに「休憩は予定どおり」UI設計指針を追加)
   - 出典: ameblo.jp/toku-422「ハンガーノック恐怖」、milbon24.com「最初の30kmで脚が重く」、知恵袋q12236541242「速度差で迷惑」

3. **初心者→中級への成長過程で必須スキルになる**
   - 当事者性: 近しい立場
   - 内容: 「3年で8割やめる」を脱するには50km→100km拡張が必要、その時必ず「グロス速度感覚」が必要。修正案A/B/Cが「中級以上専用」だと初心者→中級の卒業ロードマップが分断される
   - 修正案B粒度階層に「beginner成長期=入門グロス速度(出発/帰宅時刻のみラフ表示)」を追加要望
   - 出典: ameblo.jp/toku-422「100kmが大きな壁」、sbaa-bicycle.com 「『走れているが安全に走れていない』状態」

4. **UI翻訳案 — 「グロス速度」という言葉を初心者には見せない**
   - 当事者性: 直接の当事者(UI受け手として)
   - 内容: 時計表示「9:00 AM → 12:30 PM」、補給予定マーク、3色バーで視覚化、「グロス速度」用語不使用、「実際にかかる時間」だけで通す
   - 修正案C UI仕様提案
   - 出典: モノシル Strava★1レビュー「ルート作成めんどくさい・精度イマイチ」、persona-beginner navigation_tool

### persona-serious (シリアスレーサー) — 4個提案

1. **本番ペース配分のグロス速度は serious 当事者の最重要課題**
   - 当事者性: 直接の当事者
   - 内容: 富士ヒル・ヤビツTT・ロードレース等でのペース配分は所要時間管理そのもの。「区間ごと通過タイムをトップチューブに貼る」実践行動はグロス速度の事前モデル化
   - 修正案A強く支持。C-OVR-TIME の hypothesis 文に「競技層の本番ペース配分(区間通過タイム)」を明示要望
   - 出典: note.com/commutebymybike 富士ヒル試走「平地260W→高地230W実測」「区間通過タイム目標トップチューブ貼付」

2. **Z2 Base 練習はグロス速度ではなく「有効有酸素時間」の管理が本質**
   - 当事者性: 直接の当事者
   - 内容: Z2 Base練習の本質は「LT1以下で何時間有酸素ストレスを蓄積したか」。信号停止・補給での心拍ドロップは練習効果を毀損。グロス速度(中断込み平均速度)とは指標の方向性が**逆**
   - 修正案B「serious(練習時)=Z2 Base時間配分(グロス速度の練習版)」は誤導的、削除/修正必要
   - 出典: broadlistening_summary「信号なし・交通量少・中断なし」、Strava Fitness vs TrainingPeaks Fitness/Fatigue乖離

3. **テーパリング期は hobby 軽量グロス速度の応用で十分カバー**
   - 当事者性: 直接の当事者
   - 内容: テーパリング期(レース3週前から負荷段階低下)では「総走行時間を○時間以内に抑える」上限管理が中心。修正案Cの軽量版がそのまま運用可能
   - 修正案C強く支持(serious のテーパリング期 + hobby 層 + endurance 層の入口体験が一本化)
   - 出典: broadlistening_summary「テーパリング期短めショートインターバル軽刺激」「FTPが下がるのが怖くて休めない悪循環」

4. **hobby 軽量グロス速度は serious にも転用可能、ただし信号密度パラメータの精度が前提**
   - 当事者性: 近しい立場(hobby層提案へのserious反応)
   - 内容: 所要時間の精度はルート上の信号密度に決定的に依存。**マーカーレイヤーの「信号情報」(P-VAL第4要素)をVolt 1で同時実装** すべき
   - 修正案C実装条件として信号情報レイヤーVolt 1必須化
   - 出典: broadlistening_summary「信号多→ブレーキ・加速・精神疲労の三重苦」「尾根幹/荒川CR/霞ヶ浦一周(信号ほぼゼロ定番)」

**他層論点への直接回答**:
- endurance「Z2 Baseはグロス速度の練習版」案 → **不一致**。別軸として独立させるべき
- hobby「C-OVR-TIME に本番ペース配分統合」 → **C-OVR-TIME に統合すべき**

### persona-casual (カジュアル) — 3個提案

1. **カフェ滞在込みの所要時間が「行ける/行けない」を決める**
   - 当事者性: 直接の当事者(カジュアル層が日常言語で抱えるニーズ)
   - 内容: 「1時間走→午前コーヒー→30分走→ランチ」典型パターン。走行時間より滞在時間の合計がライド成立を左右。家族・友人と一緒で「○時に帰る」が約束済み、所要時間見積もり失敗=社会的コスト
   - 修正案C強く支持。ただし「補給/休憩」を「立ち寄り時間」「滞在時間」に翻訳要望
   - 出典: wimo.co.jp「1時間走→コーヒー→30分走→ランチ」、知恵袋q10262445820「Googleマップ早すぎる(1.5倍)」

2. **ランチ目的の「逆算到着時刻」見積もりが最大のニーズ**
   - 当事者性: 直接の当事者
   - 内容: 「○○のラーメンを13時までに食べたい」「人気店に開店11時直後に着きたい」という到着時刻の逆算が起点。営業時間・授与時間の制約あり
   - 修正案A拡張(中級以上の「出発→到着」だけでなく「到着→出発」**逆算モード**を同じエンジンで提供)。これは brevet完走見積もりとも同型で層横断機能としての普遍性が増す
   - 出典: ringringroad.com「ツール・ド・御朱印」、wimo.co.jp「行きたいカフェ駐輪可否確認」

3. **「グロス速度」という言葉自体への違和感(関係薄い立場の所感)**
   - 当事者性: 関係薄い
   - 内容: カジュアル層は「平均20km/hで走った」とすら言わない。修正案A/B/C の機能は全面賛成だが**ラベリングをカジュアル層向けに二重化**する余地がある。同じエンジン出力を「グロス速度18.5km/h」「実走時間+補給時間6.5h」「お昼12時頃ランチ着、夕方16時頃帰宅」と表示し分け
   - 修正案Cが層横断機能でVolt 1に入る場合、表示ラベルだけはプロファイル別で出し分け
   - 出典: persona-casual navigation_tool「Googleマップ単独」、daytona.co.jp「ポタリング=距離も速度も気にせず」、知恵袋q13295557828「別に頑張って速く走らなくても」

### persona-endurance (ロング/ブルベ) — 4個提案

1. **グロス速度の本質は「制限時間という外的強制があるからこそ運用される」概念であり、所要時間見積もりとは別物**
   - 当事者性: 直接の当事者
   - 内容: グロス速度(BRM 14km/h維持)は単なる「平均速度+休憩込み」ではなく、「制限時間のあるイベントで、現在のグロス速度から残り距離・残り時間を逆算し、補給/仮眠の許容予算を動的に再配分する戦略指標」。PC通過予定時刻に対する遅延量を15-25kmごとに再計算する走行中のリアルタイム意思決定の幹
   - 修正案A強く賛成(C-OVR-SUPと分離が機能する)
   - 出典: Audax Japan「平均時速15km相当」、cyclecharts.uk PBP 2023 DNF80%、cannonball24.com PBP 2023費用529,786円

2. **「軽量化されたグロス速度」(hobby向け)は概念毀損なく成立する。ただし命名と機能設計に注意が必要**
   - 当事者性: 直接の当事者
   - 内容: (a) **命名の混同回避**: hobby向け表示名は「所要時間見積もり」「実走時間予測」とし、内部設計でグロス速度の計算ロジックを共用、endurance層UI内では「グロス速度シミュ」のまま固有名詞として保持。(b) **補給/休憩想定の粒度差**: hobby向け軽量版は「コンビニ休憩X分×Y回想定」程度、endurance本来は6軸以上の複合シミュレータ。同じ計算エンジンの**入力パラメータ数を層で切り替える**設計
   - 修正案B賛成、ただし命名修正推奨
   - 出典: JSNA「30〜60g炭水化物補給」、SBAA「1〜1.5時間ごとに何か1つ摂取」、ikarugasan「50〜70kmノンストップでコンビニつなぎ」、potaride.com「400/600/1000km別仮眠戦略」

3. **serious層の「練習時のZ2 Base時間配分」をグロス速度の練習版と位置づける案には条件付き賛成**
   - 当事者性: 近しい立場
   - 内容: 概念としては成立するが用途が異なる。endurance グロス速度=制限時間内到達のための時間管理、serious Z2 Base時間配分=Z2域に何時間滞在するかのトレーニング負荷管理。共通項は「実走時間≠ペダリング時間≠目的時間」、差異は外的制約
   - 修正案B serious層自身の合意があれば賛成、なければ削除推奨(→serious本人が**不一致**と回答)
   - 出典: PBP/BRMでのPM運用者は「グロス速度」と「平均NP」を併用、persona-endurance supply_planning_practice

4. **C-OVR-SUPからグロス速度を分離した場合のendurance固有要素の再定義**
   - 当事者性: 直接の当事者
   - 内容: 修正案Aが採用された場合、既存 C-OVR-SUP を「**制限時間ペナルティ下での7軸統合計画**」と明確化。具体7軸: (1)PC通過時刻判定、(2)補給ポイント時系列マッピング、(3)仮眠予算配分、(4)GPS/ライト類のバッテリー予算、(5)雨天時のエスケープルート、(6)機材トラブル時の対応経路、(7)DNF時の輪行帰宅計画
   - 修正案Aを補強する追加提案
   - 出典: cyclecharts.uk PBP 2023 DNF80%、longride.org仮眠戦略、BikeForumsランドヌール「内蔵電池400km限界」、tourdekimamani.com Garmin拡張バッテリー落下事故
