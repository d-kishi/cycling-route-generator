## 1. 背景と目的

### プロジェクト重要目標（CLAUDE.mdに記載済み）

仮説検証型アジャイル（特に仮説キャンバス）を、Claude Codeとの協同で実践する。製造前の工程（仮説キャンバス → EARS要件定義 → ドメインモデル）をSkills等で仕組み化し、「正しいものを探す」プロセスをAIエージェントと共に回す。

### 核心的なアイデア

- セットベースからポイントベースに絞り込むプロセスを、ペルソナを持たせたAgent Teamsによる議論 + POの判断 + 想定ユーザインタビューで組み立てる
- 個人開発で最も欠けやすい「視座の多様性」をAgent Teamsで補う

## 2. 製造前プロセスの全体設計

### フェーズ構成

```
Phase 1: 仮説キャンバス作成
  ├─ 仮説の起点（根源）の議論と決定
  ├─ 初期仮説の立案（PO + 開発チームリーダー対話）
  ├─ ペルソナの設定
  └─ 仮説キャンバスの初期ドラフト作成

Phase 2: 仮説検証（セットベース → ポイントベース）
  制約: Phase 1で定めた根源から大幅に逸脱しない
  ├─ Step 1: 並列レビュー（各ペルソナが独立に評価）
  ├─ Step 2: POの参加（意見追加、対立点の明確化）
  ├─ Step 3: ディベート（対立が生じた論点のみ）
  ├─ Step 4: インタビュー結果の注入（仲間ヒアリング + SNS調査）
  └─ Step 5: MVP境界の決定（POが最終判断、必須/将来拡張/不要に分類）

Phase 3: 要件定義（/spec-requirements）
  ├─ MVP範囲のEARS形式要件化
  └─ ペルソナレビューで要件の妥当性確認

Phase 4: ドメインモデル
  └─ バウンデッドコンテキスト・エンティティ/値オブジェクトの識別

Phase 5: 実装開始（Nx初期化〜）
```

### ドキュメント体系

| レイヤー | ドキュメント | 内容 | 粒度 |
|---|---|---|---|
| 仮説 | doc/hypothesis-validation/hypothesis-canvas.yaml | 仮説キャンバス | Why / For Whom |
| 要件 | doc/requirements.md | EARS形式 + 否定的境界 | What |
| ドメイン | doc/domain-model.md | BC、エンティティ、コンテキストマップ | What/Why（構造） |
| 実装 | コード | interface定義、データモデル等 | How |

## 3. 役割分担

### Phase 1: 仮説キャンバス作成

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー（PO） | ユーザ | ビジョン・仮説の起点を提供、最終判断 |
| 開発チームリーダー | メインAgent（Claude） | 技術観点での意見、仮説の構造化支援 |

### Phase 2: 仮説検証

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー（PO） | ユーザ | 議論への参加（中級ホビーサイクリストとして）、最終判断 |
| スクラムマスター/ファシリテータ | メインAgent（Claude） | 議論の促進・整理、技術的実現性の指摘（中立だが技術判断は例外） |
| チームメンバー | ペルソナAgent（Agent Teams） | 各ペルソナの視点から仮説を評価・議論 |

### ファシリテータの行動原則

- 基本姿勢: 中立。自分の意見は混ぜない
- 例外: 技術的に実現困難な場合はPOに伝える（判断材料として）
- ペルソナAgentの意見を収集・整理し、POに分かりやすく提示する
- 対立が生じた論点を特定し、必要に応じてディベートを設計する

---

## 4. ペルソナレビューの設計

### 議論形態: 段階的アプローチ

1. **並列レビュー**: 各ペルソナAgentが独立に意見を出す
2. **ディベート**: 対立が生じた論点のみ、ペルソナAgent同士が議論
3. **POへの提示**: ファシリテータが整理してPOに判断材料を届ける

### ペルソナの設定方針

- プロジェクトのドメインに応じて自由に設定可能（汎用設計）
- スキル起動時にユーザが設定する（ハードコードしない）
- ユーザ自身が該当するペルソナも含める（自分の視点を客観化する機会）
- 推奨: 2-4体

### Agent構成

- ペルソナAgent: subagent_type: general-purpose, model: sonnet
- ファシリテータ: メインAgent（セッションのモデルを使用）

## 5. 仮説の根源（アンカー）による制約

### 課題

AIモデルは無限の情報にアクセスできるため、議論が発散しやすい。不確実な結論を出すことが本質的に難しい。

### 解決策

Phase 1で「仮説の根源」を決定し、Phase 2ではこれをアンカーとする。

- 根源は書籍の"どこから仮説を立てるべきか?"に基づき、起点（目的/課題/状況等）を議論して決定する
- Phase 2の全議論は「根源に照らして妥当か」を判断軸とする
- 根源から大幅に逸脱する提案にはフラグを立てる
- このプロジェクトの起点はPhase 1で議論して決定する（現時点では未定）

## 7. 仮説キャンバスの表現方法

### 両面アプローチ

- **Agent向け**: YAML形式の構造化文書（読み書きの合理性）
- **人間向け**: Miro MCP経由でビジュアルボードに描画（俯瞰性）

### YAML フォーマット（確定）

`doc/discussions/hypothesis-canvas-format-draft.yaml` にフォーマット案を記録済み。主要な構造:
- `meta`: プロジェクト名、バージョン、フェーズ
- `root`: 仮説の根源（Phase 2のアンカー）。type / statement / rationale
- `canvas`: 仮説キャンバス本体。customer / provider / business の3区画
  - 各仮説: `id`, `title`（一言サマリー）, `hypothesis`, `confidence`, `evidence`
- `relations`: 仮説間のリレーション（from → to, type: supports/derives/conflicts, note）

### ビジュアライズツール: Miro MCP（動作確認済み）

公式Miro MCP（`https://mcp.miro.com`）を採用。セットアップ完了済み。

**Miro選定の根拠:**
- 仮説キャンバスだけでなく、MVP特定後のイベントストーミングにも転用したい
- Miro MCPの今後の機能拡充に期待

**描画テスト結果（2026-04-03）:**
- flowchart DSLで仮説キャンバスを表現可能
- cluster機能で「顧客/ユーザ側」「提供者側」のセクション分けができる
- コネクタでリレーション（supports/derives）を表現
- 色分け: 根源=赤、顧客側=黄、提供者側=青
- テストボード: https://miro.com/app/board/uXjVGoe-c2k=/
- 書籍の構成への近づけ方は実践しながら調整する方針

**公式MCPの提供ツール**: context_explore / context_get / diagram_create / diagram_get_dsl / doc_create / doc_get / doc_update / table_create / table_list_rows / table_sync_rows / board_list_items / image_get_data / image_get_url

**フォールバック**: Miro MCPで不十分な場合、HTML生成（YAML → スタンドアロンHTML）に切り替える。
