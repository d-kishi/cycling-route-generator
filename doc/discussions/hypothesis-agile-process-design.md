## 1. 背景と目的

### プロジェクト重要目標（CLAUDE.mdに記載済み）

仮説検証型アジャイル（特に仮説キャンバス）を、Claude Codeとの協同で実践する。製造前の工程（仮説キャンバス → EARS要件定義 → ドメインモデル）をSkills等で仕組み化し、「正しいものを探す」プロセスをAIエージェントと共に回す。

### 核心的なアイデア

- セットベースからポイントベースに絞り込むプロセスを、ペルソナを持たせたAgent Teamsによる議論 + POの判断 + 想定ユーザインタビューで組み立てる
- 個人開発で最も欠けやすい「視座の多様性」をAgent Teamsで補う

## 2. 製造前プロセスの全体設計

### フェーズ構成

```
Phase 1: 仮説キャンバス作成
  ├─ 仮説の起点（根源）の議論と決定
  ├─ 初期仮説の立案（PO + 開発チームリーダー対話）
  ├─ ペルソナの設定
  └─ 仮説キャンバスの初期ドラフト作成

Phase 2: 仮説検証（セットベース → ポイントベース）
  制約: Phase 1で定めた根源から大幅に逸脱しない
  ├─ Step 1: 並列レビュー（各ペルソナが独立に評価）
  ├─ Step 2: POの参加（意見追加、対立点の明確化）
  ├─ Step 3: ディベート（対立が生じた論点のみ）
  ├─ Step 4: インタビュー結果の注入（仲間ヒアリング + SNS調査）
  └─ Step 5: MVP境界の決定（POが最終判断、必須/将来拡張/不要に分類）

Phase 3: 要件定義（/spec-requirements）
  ├─ MVP範囲のEARS形式要件化
  └─ ペルソナレビューで要件の妥当性確認

Phase 4: ドメインモデル
  └─ バウンデッドコンテキスト・エンティティ/値オブジェクトの識別

Phase 5: 実装開始（Nx初期化〜）
```

## 3. 役割分担

### Phase 1: 仮説キャンバス作成

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー（PO） | ユーザ | ビジョン・仮説の起点を提供、最終判断 |
| 開発チームリーダー | メインAgent（Claude） | 技術観点での意見、仮説の構造化支援 |

### Phase 2: 仮説検証

| 役割 | 担当 | 責務 |
|---|---|---|
| プロダクトオーナー（PO） | ユーザ | 議論への参加（中級ホビーサイクリストとして）、最終判断 |
| スクラムマスター/ファシリテータ | メインAgent（Claude） | 議論の促進・整理、技術的実現性の指摘（中立だが技術判断は例外） |
| チームメンバー | ペルソナAgent（Agent Teams） | 各ペルソナの視点から仮説を評価・議論 |

### ファシリテータの行動原則

- 基本姿勢: 中立。自分の意見は混ぜない
- 例外: 技術的に実現困難な場合はPOに伝える（判断材料として）
- ペルソナAgentの意見を収集・整理し、POに分かりやすく提示する
- 対立が生じた論点を特定し、必要に応じてディベートを設計する

## 4. ペルソナレビューの設計

### 議論形態: 段階的アプローチ

1. **並列レビュー**: 各ペルソナAgentが独立に意見を出す
2. **ディベート**: 対立が生じた論点のみ、ペルソナAgent同士が議論
3. **POへの提示**: ファシリテータが整理してPOに判断材料を届ける

### ペルソナの設定方針

- プロジェクトのドメインに応じて自由に設定可能（汎用設計）
- スキル起動時にユーザが設定する（ハードコードしない）
- ユーザ自身が該当するペルソナも含める（自分の視点を客観化する機会）
- 推奨: 2-4体

### Agent構成

- ペルソナAgent: subagent_type: general-purpose, model: sonnet
- ファシリテータ: メインAgent（セッションのモデルを使用）

## 5. 仮説の根源（アンカー）による制約

### 課題

AIモデルは無限の情報にアクセスできるため、議論が発散しやすい。不確実な結論を出すことが本質的に難しい。

### 解決策

Phase 1で「仮説の根源」を決定し、Phase 2ではこれをアンカーとする。

- 根源は書籍の"どこから仮説を立てるべきか?"に基づき、起点（目的/課題/状況等）を議論して決定する
- Phase 2の全議論は「根源に照らして妥当か」を判断軸とする
- 根源から大幅に逸脱する提案にはフラグを立てる
- このプロジェクトの起点はPhase 1で議論して決定する（現時点では未定）

## 10. 将来的なアイデア（未実装）

### /loop コマンドの活用可能性

仮説検証サイクルを「1セッションで完結させない」運用にした場合、`/loop` で定期的に新しいインタビューメモやPOコメントの追加をチェックし、次のdebateラウンドのトリガーを提案するウォッチャーとして活用できる可能性がある。まずはスキルを手動で回し、自動化したいポイントが明確になってから検討する。

### ブロードリスニングによるペルソナ形成の高度化（2026-04-14提案、2026-04-15詳細化）

ペルソナAgentに渡す情報量を増やし、よりリアルなペルソナを形成するためにブロードリスニングを活用する構想。

**アプローチ: A+Bの段階的組み合わせ**
- **A（ペルソナ形成時）**: ブロードリスニングで各ペルソナ層の言葉遣い、関心事、不満、行動パターンを収集し、ペルソナの人格・知識・特性を充実させる
- **B（debate時）**: 議論テーマに関連する情報を追加収集し、リアルな声に基づいた意見をAgentに出させる

#### ペルソナ形成スキル（`/persona`）の設計

**独立したSkill/プラグインとして設計する。** spec-hypothesisの内部機能にしない。仮説検証だけでなく、デザインレビュー、別プロジェクトへの横展開も視野に入れる。

**呼び出しパターン:**

| コマンド | 動作 |
|---|---|
| `/persona` | 新規ペルソナ形成（対話→チェックリスト→ブロードリスニング→形成） |
| `/persona update` | 既存ペルソナの更新 |
| `/persona list` | 形成済みペルソナ一覧の表示 |

**新規ペルソナ形成のワークフロー:**

1. **ユーザとの対話**: ペルソナの内容（属性、特性）とブロードリスニング用チェックリスト（収集すべき情報カテゴリ）を対話しながら形成
   - チェックリストの粒度は初回は粗く、不足時やペルソナの内容次第でLLMが自動的に細分化
2. **ファイルに保存**: ペルソナ内容とチェックリストを所定のディレクトリに保存
3. **SubAgentによるブロードリスニング**: チェックリストをpromptで渡してSubAgentを起動
4. **返却結果でペルソナを充実化**: SubAgentの要約をペルソナに統合
5. **PO承認（オプション）**: デフォルトは自動進行。初期段階では有効化を推奨

**更新時のワークフロー:**
- 既存ペルソナ + 既存チェックリストを読み込み
- 必要に応じてチェックリストをより細かい粒度に拡張
- SubAgentで追加情報収集 → ペルソナ更新

#### ハーネスエンジニアリング観点での設計見直し（2026-04-16）

Martin Fowler「Harness Engineering」(https://martinfowler.com/articles/harness-engineering.html) を参照し、Guides（フィードフォワード制御）とSensors（フィードバック制御）の観点で設計を強化する。

**核心的な問題点:**
- 当初設計ではGenerator/Evaluator分離がなく、同じSubAgentが「収集」と「十分性評価」を兼ねていた → 確証バイアスのリスク
- Sensors（検証機構）がほぼ未設計 → 品質の下限が保証されない
- Guides（行動前の方向付け）が暗黙的 → SubAgentが汎用的でありながら拠り所が不足

これらを以下の3点で改善する:
1. Generator/Evaluator分離
2. Guidesの明文化（SKILL.md本体に記述）
3. 計算的Sensorsの追加（LLM判断に依存しない決定論的検証）

#### Generator/Evaluator分離の構成

2つのSubAgentに分離する:

```
メインAgent
  │
  ├─ Collector SubAgent（収集役）
  │   - ブロードリスニングで情報収集
  │   - 要約の生成
  │   - チェックリスト全カテゴリの要約を返却
  │
  ├─ Evaluator SubAgent（評価役）
  │   - Collectorの成果物を受け取り
  │   - 品質評価（情報源の多様性、バイアス、具体性等）
  │   - 不十分と判断した場合は追加収集の方針を提示
  │
  └─ メインAgent
      - Evaluatorの判定に基づき再度Collectorを呼ぶか判断
      - 合格ならペルソナ充実化に進む
```

**Collectorの完了条件（客観的に検証可能）:**
- 渡されたチェックリストの全カテゴリについて要約を生成した
- 各カテゴリで使用した情報源を記録した
- 各カテゴリの自己評価（参考程度）を付記した

**Evaluatorの判定観点:**
- 情報源の多様性（1種類のソースに偏っていないか）
- 具体性（抽象論ではなく具体的なエピソード・声があるか）
- 一次情報の含有率
- カテゴリ間の情報バランス

**ループ制御:**
- Collector → Evaluator のループは最大3回まで
- それでも不十分な場合はメインAgentがPOに判断を仰ぐ

**200kコンテキスト制限への対策:**
- Collector側で「収集→即座に要約→生データ破棄→次の収集」の逐次処理

**情報の仕切り:**
- ペルソナ別にCollector/Evaluatorペアを分離起動するため、情報汚染がアーキテクチャレベルで防止される

**memory設定:**
- Collector/Evaluator両SubAgentに `memory: user` スコープを付与する
- 理由: メタ知識（情報の見つかりやすさ、評価基準の精緻化等）はドメイン非依存であり、Plugin公開時に利用者の学習が継続する

#### Guidesの明文化（SKILL.md本体に記述する内容）

SubAgentはreferences/を読まないため、核心となるGuidesはSKILL.md本体に直接記述する。

**「良いペルソナの基準」（6項目）:**

1. **明確な差別化**: 他のペルソナと役割・視点が重複していない
2. **具体性**: 抽象的でない（「サイクリスト」ではなく「週末に30km走るロード初心者」のレベル）
3. **一次情報の反映**:
   - 実在するユーザ層から得られる声・データに基づいて形成する
   - 推測や想像だけで構築しない
   - 直接的な一次情報が得られない場合:
     - 著名な発信者（YouTuber、ブロガー等）を参照することは可。ただし1人に依存せず、複数の発信者や情報源から要素を抽出する
     - 抽出した要素は統合・抽象化し、特定個人のコピーではなく「その層を代表するアーキタイプ」として形成する
     - 発信者は平均的ユーザではない（発信力のある特殊層）ことを認識し、そのバイアスを補正する
4. **内部整合性**: 属性間に矛盾がない（例: 「初心者」と「レース経験豊富」の併存は不可）
5. **ドメイン適合性**: プロジェクトの文脈で意味のある視点を提供できる
6. **情報源の多様性**: 複数の情報源から形成され、1つに偏らない

**ブロードリスニングの情報源優先順位:**

| 優先度 | 情報源 | 特徴 |
|---|---|---|
| 高 | 公式・一次情報（公式サイト、公的調査、専門メディア） | 信頼性最高 |
| 中 | 専門コミュニティ（Reddit特定スレッド、専門フォーラム） | 実ユーザーの声、濃い情報 |
| 中 | SNS（X、Strava等、日本語圏含む） | リアルタイム性、ユーザの本音 |
| 低 | 一般ブログ、個人ブログ | バイアスあり、補助的 |
| 除外 | アフィリエイトサイト、SEO対策コンテンツ | ノイズが多い |

#### 計算的Sensors（決定論的検証）

LLM判断に依存しない機械的な品質下限を保証する。

| Sensor | 対象 | 配置タイミング | 失敗時の挙動 |
|---|---|---|---|
| ペルソナID重複チェック | ペルソナYAML | 新規作成前（ファイル保存前） | エラー表示、ID変更を求める |
| YAMLスキーマ検証 | ペルソナYAML | ファイル保存直後 | エラー表示、手動修正 |
| チェックリスト完全性 | Collector成果物 | Collector返却直後 | 不足カテゴリを提示、再収集 |
| 情報源URL有効性 | Collector成果物 | Evaluator前 | 死にリンクをフラグ、警告のみ |
| 必須フィールド有無 | ペルソナYAML | 充実化後 | エラー表示、手動修正 |

1人のエンジニアの視野を拡大し、多様性のある視点を持ち込める仕組みとして、Claude Codeプラグインとしての公開も視野に入れている。「コードを書くAgent」ではなく「多様な視点を持つペルソナAgent」という希少な価値提供になりうる。
